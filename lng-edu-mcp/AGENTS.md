# AGENTS.md

## Цель проекта

Создать учебную платформу для сербского и английского языков. MCP-сервер хранит книги, прогресс чтения, словарь, ошибки, интервальные повторения и статистику. ChatGPT и web-приложение являются клиентами; состояние обучения нельзя хранить только в prompt или истории чата.

## Структура

- `backend/` — Java 21, Spring Boot, Gradle, PostgreSQL, MCP endpoint и REST API.
- `web/` — Vue 3, TypeScript, Vite.
- `docs/` — архитектура, ADR и API-документация.
- `prompts/` — задания агентам и системные prompts для учебного ассистента.
- `infra/` — локальная инфраструктура и deployment-конфигурация.

Перед изменениями прочитай этот файл, `README.md`, релевантный prompt и документацию в `docs/`.

## llm-wiki-plus

Документация ведётся как LLM-читаемая wiki с тегами, связывающими код и документацию. Реестр тегов — [`docs/tags.md`](docs/tags.md). Соблюдай на каждой задаче:

- **Документация — это LLM-wiki.** `index.md` хранятся **только в `docs/`** — в каталогах кода и в корне проекта их создавать не нужно. Корень wiki — [`docs/index.md`](docs/index.md); каждый значимый каталог документации имеет `index.md` с однострочным описанием каждого файла и подкаталога, с двунаправленными ссылками (родитель ↔ дети). Каждую деталь описывает одна страница.
- **Читай перед действием.** Перед задачей пройди дерево `index.md` в `docs/` до страниц, релевантных коду, который будешь менять.
- **Обновляй по ходу.** В том же изменении обновляй затронутые страницы `docs/` и их `index.md` (добавленные, перемещённые, удалённые или перепрофилированные файлы и концепции).
- **Тегируй сквозные концепции.** Когда концепция охватывает код и документацию, выбери slug, зарегистрируй его в `docs/tags.md` и размести токен `@tag:<slug>` в обоих местах: строкой у начала `.md`-страницы в `docs/` (для каталога документации — в его `index.md`) и комментарием у начала файла кода (`// @tag:<slug>`, `# @tag:<slug>`, `/* @tag:<slug> */`; для каталога кода — в ведущем комментарии его основного модуля). Slug — строчный kebab-case `[a-z0-9-]+`; несколько тегов пиши через пробел. Не выдумывай теги для разовых деталей.

Поиск тегов:

```bash
grep -rn "@tag:<slug>" .                     # все места с тегом
grep -oE "@tag:[a-z0-9-]+" path/to/file      # теги файла
grep -rhoE "@tag:[a-z0-9-]+" . | sort -u     # все теги репозитория
```

## Инженерные правила

1. Сначала реализуй минимальный вертикальный сценарий: пользователь начинает сессию, получает следующий фрагмент, фиксирует понимание/неизвестное слово, завершает сессию и видит статистику.
2. PostgreSQL — источник истины. Tool-вызовы должны быть идемпотентными там, где повтор запроса может создать дубль.
3. MCP tools делай небольшими, явно типизированными и пригодными для LLM. Не смешивай чтение данных и запись нескольких несвязанных результатов.
4. REST API предназначен для web-клиента; MCP и REST используют один application/service layer.
5. Не доверяй `user_id`, оценкам модели и произвольному JSON без валидации. До появления аутентификации используй явный локальный профиль, не имитируй production security.
6. Миграции БД выполняй только через Flyway. Не включай Hibernate schema auto-update.
7. Время храни в UTC, язык — BCP 47 (`sr`, `en`), пользовательские даты рассчитывай в его timezone.
8. Не сохраняй секреты, книги или персональные данные в git. Для книг учитывай авторские права.
9. Новые решения с существенными trade-off фиксируй коротким ADR в `docs/adr/`.
10. Не добавляй инфраструктуру и абстракции, которые не нужны текущему вертикальному сценарию.

## Backend

- Package-by-feature; слои внутри feature: transport/application/domain/persistence по необходимости.
- Spring Boot stable release, Java 21, Gradle Wrapper и Kotlin DSL.
- Spring Web, Validation, Data JPA, Flyway, PostgreSQL driver, Actuator; MCP SDK/стартер выбирать по актуальной официальной документации.
- API errors — единый Problem Details формат.
- Интеграционные тесты репозиториев и миграций — Testcontainers PostgreSQL.
- Unit-тесты — JUnit 5; внешние модели/LLM не требуются для доменной логики.
- Добавляй OpenAPI для REST и JSON Schema/описания для MCP tools.

Проверка перед завершением: `./gradlew :backend:check`.

## Web

- Vue 3 Composition API, TypeScript strict, Vite, Vue Router, Pinia.
- HTTP-клиент и DTO централизованы; желательно генерировать типы из OpenAPI после стабилизации API.
- Состояние сервера не дублируй без необходимости в глобальном store.
- Интерфейс сначала ориентирован на desktop/mobile browser и доступность с клавиатуры.
- Для основных экранов добавляй component/unit tests; критический учебный сценарий покрывай e2e после появления стабильного API.

Проверка перед завершением: `npm run lint && npm run typecheck && npm test -- --run && npm run build`.

## Git и качество

- Не меняй несвязанные файлы.
- Коммиты должны быть небольшими и описывать один законченный шаг.
- Перед добавлением зависимости объясни ее назначение и проверь актуальную стабильную версию.
- Используй Gradle Version Catalog для общего управления версиями, когда появится больше одного потребителя зависимости; не дублируй версии по build-файлам.
- В README поддерживай точные команды запуска и список обязательных переменных окружения.
- Если требование неоднозначно, зафиксируй разумное допущение в `docs/decisions.md` и продолжай; спрашивай пользователя только при дорогом или необратимом выборе.
